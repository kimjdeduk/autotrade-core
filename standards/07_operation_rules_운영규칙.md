# 운영 규칙

이 프로젝트의 운영 규칙은 아래와 같다.

## 1. 큰 작업 한 번에 금지
- 전체 시스템을 한 번에 만들려고 하지 않는다.
- 기능은 작은 단위로 나눠서 만든다.
- 한 번에 하나의 기능만 만들고 검증한다.

## 2. 작업 순서 고정
기본 순서는 아래와 같다.
1. 구조 설계
2. 파일 생성
3. 단독 실행 테스트
4. 오류 수정
5. 결과 저장 확인
6. 보고 형식 정리
7. OpenClaw 연결

## 3. 시스템 구축과 자동매매 전략 작업 분리
- 시스템 구축 채팅과 자동매매 전략 채팅을 분리한다.
- 구조 문제와 전략 문제를 한 번에 섞지 않는다.

## 4. 오류 수정은 별도 확인
- 오류가 나면 먼저 에러 로그를 확인한다.
- 정상 실행 전에는 다음 단계로 넘어가지 않는다.
- 추측 수정 대신 로그 기반 수정 원칙을 따른다.

## 5. OpenClaw 역할 제한
- OpenClaw 안에서는 무거운 본체 개발을 하지 않는다.
- OpenClaw는 실행, 검수, 요약, 보고 중심으로 사용한다.
- 계산은 밖에서 하고 정리는 안에서 한다.

## 6. 결과 파일 우선 원칙
- 실행 결과는 outputs 폴더에 저장한다.
- OpenClaw는 결과 파일을 읽어 정리한다.
- 화면 출력만 하고 끝나는 구조는 피한다.

## 7. 문서화 원칙
- 중요한 결정사항은 문서로 남긴다.
- 역할 변경, 구조 변경, 파일 규칙 변경은 즉시 기록한다.
- 나중에 기억에 의존하지 않는다.

## 8. 테스트 우선 원칙
- 새 기능은 바로 실전 연결하지 않는다.
- 먼저 단독 실행
- 그다음 테스트
- 그다음 백테스트
- 마지막에 OpenClaw 연결 순서로 진행한다.

## 9. 실주문 연결 보류 원칙
- 시스템 구축 단계에서는 실주문 연결을 최우선으로 두지 않는다.
- 실주문은 구조와 검증이 충분히 안정된 뒤 마지막에 연결한다.

## 10. 대화 운영 원칙
- 긴 작업은 새 채팅으로 나눈다.
- 한 채팅에서는 한 주제만 다룬다.
- 중간 결정사항은 요약 문서에 반영한다.
