# 기존 자산 분류

이 문서는 현재 이미 가지고 있는 스킬, 코드, 문서, 설정을
앞으로 어떤 방식으로 재활용할지 분류하기 위한 기준 문서다.

## 1. 분류 목적
기존 자산을 전부 새로 만들지 않고,
아래 3가지 기준으로 나눈다.

1. 그대로 사용
2. 수정 후 사용
3. 분리 또는 폐기 검토

이 분류를 통해
- 재활용 가능한 것은 살리고
- 구조에 맞지 않는 것은 정리하고
- 앞으로 autotrade-core와 OpenClaw 역할을 분리한다.

## 2. 그대로 사용
다음 유형은 OpenClaw 안에서 그대로 재활용 가능하다.

- 로그 요약 스킬
- 결과 보고서 정리 스킬
- 텔레그램 전송 스킬
- 오류 요약 스킬
- 검수 보고 형식
- 역할별 보고 템플릿
- 상태 점검 보고 템플릿

### 기준
- 계산 본체가 아니다.
- 운영, 검수, 보고, 알림 중심이다.
- 결과를 정리해서 전달하는 역할이다.

## 3. 수정 후 사용
다음 유형은 그대로 쓰지 말고 구조를 바꿔서 재사용한다.

- 데이터 수집 스킬
- 전략 계산 스킬
- 종목 후보 생성 스킬
- 백테스트 스킬
- 리포트 생성 스킬
- 실행 스크립트 초안
- 파일 저장 로직

### 기준
- 현재는 OpenClaw 안에서 직접 계산하거나 처리하고 있다.
- 앞으로는 계산 본체를 autotrade-core로 옮기고
  OpenClaw에는 실행, 결과 읽기, 요약 규칙만 남긴다.

### 수정 방향
- 실제 계산 로직은 autotrade-core로 이동
- OpenClaw 스킬은 실행 지시와 결과 정리 역할만 유지
- 출력은 outputs 폴더 기준으로 저장

## 4. 분리 또는 폐기 검토
다음 유형은 바로 재활용하지 말고 먼저 검토한다.

- 한 파일에 여러 역할이 섞인 올인원 스킬
- 데이터 수집 + 계산 + 보고가 한 번에 들어간 구조
- 경로가 하드코딩된 파일
- 특정 세션 구조에 강하게 묶인 스킬
- 실주문까지 한 번에 처리하는 스킬
- 너무 긴 프롬프트형 운영 문서
- 역할이 불분명한 옛 규칙 문서

### 기준
- 현재 구조와 맞지 않는다.
- 수정 비용이 너무 크다.
- 나중에 다시 꼬일 가능성이 높다.

## 6. 분류 원칙
- 운영/검수/보고 기능은 OpenClaw에 남긴다.
- 계산/수집/전략/백테스트 본체는 autotrade-core로 옮긴다.
- 결과물은 outputs 기준으로 저장한다.
- 역할이 여러 개 섞인 자산은 분리 우선 원칙을 적용한다.
- 헷갈리면 일단 수정 후 사용으로 분류하고 검토한다.

## 7. 현재 프로젝트 적용 원칙
- 지금은 자동매매 실행보다 시스템 구축이 우선이다.
- 따라서 기존 자산도 실주문 중심이 아니라
  구조 정리와 재활용 기준 확립 중심으로 분류한다.
- 바로 쓰는 것보다 오래 유지 가능한 구조를 우선한다.

## 8. 한 줄 요약
기존 자산은 전부 버리지 않는다.
하지만 그대로 다 쓰지도 않는다.
본체 기능은 autotrade-core로,
운영 기능은 OpenClaw로 나눠서 재활용한다.
