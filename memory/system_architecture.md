# 시스템 아키텍처 및 환경 구조

이 문서는 Commander(김재덕)의 자동매매 시스템 프로젝트(autotrade-core)의 전반적인 시스템 아키텍처와 환경 구조를 정의합니다. 사자(Manus)는 매 세션 시작 시 이 문서를 참조하여 시스템 구조를 파악해야 합니다.

## 1. 물리적 구조 및 환경

시스템은 Hostinger VPS 환경에서 구동되며, Windows 11 기반의 WSL(Ubuntu) 환경에서 실행됩니다. 핵심 컴포넌트인 OpenClaw Gateway는 포트 18789에서 systemd 서비스로 동작합니다. 물리적으로는 단일 에이전트(deepseek_system)와 단일 모델(DeepSeek Chat/V3)로 구성되어 있으나, 프롬프트 시뮬레이션을 통해 여러 페르소나(아사, 아싸, 아자, 아짜)를 구현합니다.

## 2. 텔레그램 봇 연동

시스템은 텔레그램 봇을 통해 사용자와 상호작용합니다. 주요 봇의 역할은 다음과 같습니다.

| 봇 이름 | 봇 ID | 역할 |
|---|---|---|
| 아사 | @assa_openclaw_bot | OpenClaw 메인 인터페이스, 작업 지시 수신 및 실행 |
| 알림 | @Alarm1072_bot | 시스템 이벤트, 에러, 작업 완료 등의 자동 알림 전송 |

## 3. 사자(Manus)와 아사(OpenClaw)의 협업 구조

사자(Manus)는 프로젝트를 총괄하는 AI 참모장 역할을 수행하며, 아사(OpenClaw)는 실제 코드를 작성하고 실행하는 실무자 역할을 담당합니다. 두 AI 간의 협업은 GitHub 저장소를 매개로 이루어집니다.

사자는 작업 지시문을 작성하여 GitHub의 `instructions/` 디렉토리에 push합니다. 이후 Commander가 아사에게 "pull 받고 실행해"라는 단일 명령을 내리면, 아사는 GitHub에서 지시문을 pull하여 읽고 작업을 수행합니다. 작업이 완료되면 아사는 결과 코드를 다시 GitHub에 push하고, 알림봇을 통해 완료 메시지를 전송합니다. 사자는 GitHub에서 결과 코드를 직접 pull하여 검수하고, 텔레그램 Bot API를 통해 알림 메시지를 모니터링합니다. 이 구조를 통해 Commander의 복사-붙여넣기 작업을 최소화합니다.

## 4. 아사(OpenClaw) 내부 협업 규칙

아사 내부에서는 5단계의 협업 프로세스를 거쳐 결과물을 도출합니다.

1. **아사 (초안 작성):** 요구사항을 분석하고 초기 코드 초안을 작성합니다.
2. **아싸 (논리 추론):** 작성된 초안의 논리적 오류를 검증하고 개선 방안을 도출합니다.
3. **아자 (코드 정리):** 검증된 로직을 바탕으로 코드를 리팩토링하고 주석을 추가합니다.
4. **아짜 (최종 검수):** 완성된 코드가 요구사항을 모두 충족하는지 최종적으로 확인합니다.
5. **사자 (최종 승인):** 아짜의 검수를 통과한 코드를 사자가 최종적으로 승인하고 GitHub에 반영합니다.

## 5. 보안 및 운영 원칙

시스템 운영에 있어 다음의 보안 및 운영 원칙을 엄격히 준수해야 합니다.

*   **민감 정보 관리:** API 키, 토큰, 계좌번호 등 모든 민감 정보는 코드 내 하드코딩을 금지하며, 반드시 `.env` 파일을 통해 관리해야 합니다.
*   **정보 노출 방지:** 로그, 알림 메시지, GitHub 커밋 메시지 등에 실제 비밀값이 출력되지 않도록 주의해야 합니다.
*   **단계적 실행:** 한 번에 여러 단계를 묶어서 진행하는 것을 지양하고, 설정, 확인, 테스트, 저장, 알림, 커밋의 순서로 단계별로 진행해야 합니다.
*   **버전 관리:** `.env` 파일 및 로컬 데이터 파일은 `.gitignore`에 포함하여 GitHub에 커밋되지 않도록 차단해야 합니다.
*   **사용자 승인:** GitHub push 등 시스템 상태를 변경하는 주요 작업은 반드시 Commander의 승인 하에 진행해야 합니다.
